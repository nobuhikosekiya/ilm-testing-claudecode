## Scenario intent

Tests the hypothesis that adding `wait_for_snapshot` to the ILM delete phase (pointing at an SLM policy) protects Frozen-tier backing indices before deletion. The scenario demonstrates that this assumption is **wrong in two compounding ways**.

### ILM policy

| Phase | Setting |
|---|---|
| Hot | `rollover: max_docs: 1` |
| Frozen | `min_age: 0m` → searchable snapshot to `found-snapshots` |
| Delete | `min_age: 3m` → `wait_for_snapshot: { policy: approach_a_wfs_slm_policy }` then `delete: { delete_searchable_snapshot: false }` |

### SLM policy

| Setting | Value |
|---|---|
| Schedule | `0 0 0 1 1 ? 2099` (never fires automatically) |
| `max_count` | 3 (retention deletes oldest when exceeded) |
| `partial` | `true` (allows snapshotting frozen mounts) |

## Findings

### Finding 0 — SLM policy must be created before the ILM policy

When attempting to `PUT _ilm/policy/approach_a_wfs_policy` before the SLM policy existed, Elasticsearch returned HTTP 400:

```json
{
  "error": {
    "type": "illegal_argument_exception",
    "reason": "no such snapshot lifecycle policy [approach_a_wfs_slm_policy], the snapshot lifecycle policy referenced by the [wait_for_snapshot] action in the [delete] phase must exist before it can be referenced by an ILM policy"
  }
}
```

**Implication:** In the scenario file, the SLM policy (Step 2.5) must be created before the ILM policy (Step 2). The scenario file has been updated accordingly.

---

### Failure 1 — Permanent ILM stall ❌

`wait_for_snapshot` requires the SLM policy to complete a successful snapshot **after** the index entered the delete phase. The last automatic SLM execution was snapshot 6, completed before the delete phase was entered at 11:45:59 UTC. With SLM scheduled for year 2099, no new snapshot will ever fire automatically.

**ILM explain for `partial-.ds-logs-mytest-wfs-default-2026.05.27-000001`:**

```json
{
  "phase": "delete",
  "action": "wait_for_snapshot",
  "step": "wait-for-snapshot",
  "step_info": {
    "message": "waiting for policy 'approach_a_wfs_slm_policy' to be executed since Wed May 27 11:45:59 GMT 2026"
  }
}
```

All **three** frozen backing indices (000001, 000002, 000003) were simultaneously stuck. The stall was confirmed over multiple polling ticks with no progression. In production, this means:

- Frozen mounts remain alive indefinitely, consuming cache and heap
- The data stream never advances cleanly
- No cleanup occurs until a human intervenes
- The stall window scales with SLM's schedule interval (hourly → up to 1h stall; nightly → up to 24h; weekly → up to 7 days)

#### Restore attempt from snapshot 7

| Restore target name | Outcome |
|---|---|
| `.ds-logs-mytest-wfs-default-2026.05.27-000001` | **404 index_not_found** — snapshot 7 never contained this index name |
| `partial-.ds-logs-mytest-wfs-default-2026.05.27-000001` | **1 doc returned — but only conditionally** (see below) |

#### The conditional success — and why it reveals the deeper trap

Restoring using `partial-.ds-...` succeeded and returned 1 document **only because `delete_searchable_snapshot: false` kept the underlying ILM snapshot alive.** The chain was:

```
SLM snapshot 7 → partial mount entry → ILM snapshot (still present) → segment data
```

This is not independent protection. The SLM snapshot is a **pointer into the ILM snapshot**, not a copy of the data.

| Scenario | `delete_searchable_snapshot` | Restore from SLM snapshot 7 |
|---|---|---|
| This test | `false` (ILM snapshot kept) | ✅ 1 doc — but only via ILM snapshot |
| Default / production default | `true` (ILM snapshot deleted) | ❌ 0 docs or restore error — no backing data |

**If the ILM snapshot exists → you don't need the SLM snapshot. If the ILM snapshot is gone → the SLM snapshot is useless too.**

## Key takeaways

1. **`wait_for_snapshot` + frozen tier + infrequent SLM = permanent ILM stall.** The delete phase will be stuck for as long as SLM's schedule interval. With a nightly SLM schedule, every frozen backing index stalls the data stream for up to 24 hours on every rollover cycle.

2. **SLM snapshots of frozen indices are not independent backups.** By the time `wait_for_snapshot` fires in the delete phase, the backing index has been a frozen (partial) mount for at least `delete.min_age`. Any SLM snapshot taken at or after that point captures only the mount metadata. The snapshots that held the original segment data have typically already been rotated out by `max_count` retention.

3. **The "protection" is circular.** A successful SLM restore from a partial-mount snapshot depends entirely on the ILM snapshot being present. `delete_searchable_snapshot: false` already provides that protection directly — `wait_for_snapshot` adds nothing.

4. **The scenario also revealed a creation-order requirement:** the SLM policy must exist before the ILM policy that references it via `wait_for_snapshot`.

5. **Correct approach:** use `delete_searchable_snapshot: false` to retain the ILM snapshot. Do **not** add `wait_for_snapshot` in a frozen-tier ILM delete phase.

## Notes for the scenario file

- Step 2 and Step 2.5 order must be swapped: **create the SLM policy (Step 2.5) before the ILM policy (Step 2)**.
- `_ilm/explain` does not accept the original `.ds-...` index name for frozen mounts — use `partial-.ds-..._ilm/explain?expand_wildcards=all` or the wildcard form `partial-.ds-logs-mytest-wfs-*/_ilm/explain?expand_wildcards=all`.
- By snapshot 3 (taken during the 20s ILM rollover wait after doc 3), index 000001 had **already been frozen** — ILM moved it to frozen tier faster than expected. Consider shorter waits or inline ILM explain checks to catch the freeze event more precisely in future runs.
