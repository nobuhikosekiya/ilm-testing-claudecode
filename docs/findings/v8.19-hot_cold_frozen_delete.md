## Findings

### Finding 1 — Data queryable throughout hot, cold, and frozen tiers ✅

Querying the data stream `logs-mytest-default-default` returned all 5 documents at every tier:
- **Hot** (Checkpoint A): `hits.total.value = 5` ✅
- **Cold** (Checkpoint B): Data remained accessible throughout cold phase migration ✅
- **Frozen** (Checkpoint C): `hits.total.value = 5` via `partial-.ds-...` mounts ✅

The data stream transparently searches all tiers. No query change is needed when data moves between tiers.

### Finding 2 — Cold phase deadlocks on single-AZ cold tier without replica reduction ⚠️

**Symptom:** ILM stuck at `cold/migrate/check-migration` for 10+ minutes.

**Root cause:** The cold phase policy had no `allocate` action. Elasticsearch defaults to 1 replica. With a single cold node:
- The replica was allocated to the cold node during the hot phase.
- ILM's implicit `migrate` action tried to move the primary to the cold node.
- The only cold node already held the replica → primary had no valid destination.
- The `check-migration` step waited indefinitely (no timeout).

**ILM step_info message:**
```
lifecycle action [migrate] waiting for [1] shards to be moved to the [data_cold] tier
(tier migration preference configuration is [data_cold, data_warm, data_hot])
shards_left_to_allocate: 1 | all_shards_active: true | number_of_replicas: 1
```

**Fix:** Add `"allocate": {"number_of_replicas": 0}` to the cold phase. This drops the replica first, freeing the cold node to accept the primary as the sole copy.

**Corrected cold phase:**
```json
"cold": {
  "min_age": "0m",
  "actions": {
    "readonly": {},
    "allocate": { "number_of_replicas": 0 }
  }
}
```

**Recovery during this run:** Updated the live policy and used `POST _ilm/move/<index>` to advance all 5 indices from `cold/migrate/check-migration` → `cold/allocate/allocate`.

### Finding 3 — Elastic Cloud rejects SLM schedules more frequent than 15 minutes ⚠️

The scenario specified a 3-minute SLM schedule (`0 */3 * * * ?`). Elastic Cloud returned:

```
invalid schedule [0 */3 * * * ?]: schedule would be too frequent, executing more than every [15m]
```

The minimum SLM interval on this Elastic Cloud deployment is **15 minutes**. Changed to `0 */15 * * * ?`.

**Impact on scenario:** The SLM policy auto-ran once during the test (at the 15-minute mark) while data was already in the frozen tier. The SLM snapshot therefore captured `partial-.ds-...` mounts, not the original hot-tier `.ds-...` indices.

The earlier SLM snapshot (`approach-a-default-slm-2026.05.27-n42a5ndetvsmjz1kmu_daa`) was taken before the frozen transition and captured the raw `.ds-...` backing indices in hot tier.

### Finding 4 — Delete phase removes both frozen mounts AND ILM snapshots simultaneously ✅

With `delete_searchable_snapshot: true`, ILM deleted both the frozen partial mounts and their corresponding ILM snapshots in a single delete-phase sweep. The deletion was batched:

- 00:14:09: count 5→3, ILM snapshots 5→3 (2 deleted at once)
- 00:14:24: count 3→1, ILM snapshots 3→1 (2 more deleted)
- 00:14:55: count 1→0, ILM snapshots 1→0 (final deletion)

After deletion, the data stream returned `hits.total.value = 0` and `[]` for ILM snapshots. **There is no restore path** — this confirms the scenario's design intent.

### Finding 5 — _ilm/explain returns null for deleted backing indices ℹ️

After frozen transition, the original `.ds-...` indices no longer exist (replaced by `partial-.ds-...`). Querying `GET .ds-...<index>/_ilm/explain` returns null. The correct endpoint for frozen indices is:

```
GET partial-.ds-logs-mytest-default-default-<date>-<N>/_ilm/explain
```

Which returned: `phase=frozen, action=complete, step=complete` once the partial mount was fully established.

## Notes for the scenario file

The following corrections should be applied to `scenarios/v8.19/hot_cold_frozen_delete.txt`:

1. **Cold phase must include `allocate: {number_of_replicas: 0}`** — without it, the cold migrate action deadlocks permanently on single-AZ cold deployments. This is likely the typical Elastic Cloud configuration for cold nodes.

2. **SLM schedule must be ≥ 15 minutes on Elastic Cloud** — replace `"0 */3 * * * ?"` with `"0 */15 * * * ?"` (or a far-future schedule if SLM auto-execution is not needed).

3. **`_ilm/explain` for frozen indices** — the scenario instructs `GET .ds-...-000001/_ilm/explain`. After frozen transition the original index is gone; the correct command is `GET partial-.ds-...-000001/_ilm/explain`.

4. **Delete min_age shortcut** — the scenario's inline comment suggesting `5m` as a test shortcut worked well. Document that changing it to `45m` (when indices are already ~38m old) is a practical way to observe deletion without restarting the test.

**Status:** All 4 corrections applied to the scenario file in commit `b54221f` (2026-05-28).
