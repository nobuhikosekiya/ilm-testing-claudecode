### Findings

**Finding 1 ‚Äî Monitoring loop: `_cat/indices` pattern misses frozen mounts**
`_cat/indices/.ds-logs-mytest-*` does not match `partial-.ds-logs-mytest-*` frozen-tier mounts. The Step 5 loop exited on its first poll thinking the indices were deleted when they were actually in the frozen phase. Correct check is `GET _data_stream/logs-mytest-default` and inspecting `.data_streams[0].indices[].index_name` for the absence of `000001`/`000002`.

**Finding 2 ‚Äî RESTORE_PREFIX mismatch between script and scenario file**
`scripts/v8.19/restore_index.sh` uses `RESTORE_PREFIX="restored2-"`, producing index names like `restored2-logs-mytest-default-2026.05.28-000001`. The scenario file `host_frozen_delete_keepsnapshot.txt` still shows `restored-` in all examples. The script is authoritative; the scenario file needs updating.

**Finding 3 ‚Äî ILM moves to frozen tier faster than 20s ingest waits**
During the 20s waits between ingestion steps, ILM already moved 000001 and 000002 to frozen. By the time the Step 5 monitoring loop started, all three backing indices were already `partial-.ds-...` frozen mounts and an extra rollover to 000004‚Üí000005 had occurred (generation 7).

**Finding 4 ‚Äî Restore from ILM snapshot confirmed ‚úÖ**
Both 000001 and 000002 restored successfully via `restore_index.sh`:
- Snapshot lookup by `.indices[]` content works correctly
- `rename_pattern: "\\.ds-(.+)"` ‚Üí `restored2-$1` produced correct index names
- Recovery completed in <200ms per index (snapshot ‚Üí hot tier)
- Wildcard search `restored2-logs-mytest-*/_search` returned 2 docs with correct content

### Findings

**Finding 1 ‚Äî ILM completed full frozen‚Üídelete cycle in <100 seconds from ingestion**

**Finding 2 ‚Äî `_cat/indices/.ds-logs-mytest-default-*` approach worked correctly this time**
Unlike Run 4 where this pattern caused a false-positive exit (indices were frozen, not deleted), in this run the indices were genuinely deleted when the loop checked. **Finding 3 ‚Äî Extra generation rollover: 000004‚Üí000005 observed**
The active write index was 000005 (0 docs), with 000003 still visible as a hot-tier index (1 doc). This extra rollover (000004‚Üí000005) is consistent with the ILM searchable_snapshot action triggering an additional rollover of the write index before mounting 000003 as frozen, as observed in Run 4 and Run 5.

Using `cleanup_old_snapshots.sh mytest 30` returned 0 results because the test snapshots were only ~3‚Äì5 minutes old.
