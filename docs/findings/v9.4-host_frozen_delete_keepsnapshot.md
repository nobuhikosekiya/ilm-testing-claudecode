## Findings & Fixes Applied to `test_scenario.txt`

### Fix 1 — Index Template Priority (Step 3)

**Problem:** The original template had no `priority` field (default = 0). The built-in `logs` template has `priority: 100` and pattern `logs-*-*`, which also matches `logs-mytest-default`. Elasticsearch applied the higher-priority built-in template, so the data stream was governed by the built-in **`logs`** ILM policy — not `approach_a_ds_policy`. No rollover occurred and all docs landed in `000001`.

**Fix:** Added `"priority": 200` to `approach_a_ds_template`.

```json
PUT _index_template/approach_a_ds_template
{
  "index_patterns": ["logs-mytest-*"],
  "priority": 200,          // ← must be > 100 to beat built-in "logs" template
  "data_stream": {},
  "template": {
    "settings": {
      "index.lifecycle.name": "approach_a_ds_policy"
    }
  }
}
```

---

### Fix 2 — ILM Frozen Snapshot Index Naming (`fm-clone-`) (Step 6)

**Problem:** The original restore command used `"indices": ".ds-logs-mytest-default-...-000001"`. However, when ILM's frozen phase snapshots a backing index, it first force-merges the index into a clone with an `fm-clone-<uuid>-` prefix, then snapshots that clone. The actual index name inside the ILM snapshot is:

```
fm-clone-9yz2-.ds-logs-mytest-default-2026.05.27-000001
```

Using the bare `.ds-` name in the restore would fail with "index not found in snapshot".

**Fix:** Updated the restore command to use the `fm-clone-` index name and a flexible rename pattern:

```json
POST _snapshot/found-snapshots/<snapshot_name>/_restore
{
  "indices": "<fm-clone-uuid>-.ds-logs-mytest-default-...",
  "rename_pattern": ".*\\.ds-(.+)",       // strips fm-clone-<uuid>- prefix
  "rename_replacement": "restored-$1",
  "index_settings": {
    "index.lifecycle.name": null,
    "index.routing.allocation.include._tier_preference": "data_hot"
  }
}
```

To find the exact `fm-clone-` name, inspect the snapshot:
```
GET _snapshot/found-snapshots/<ilm_snapshot_name>
```
and look at the `indices` array.

### Fix 3 — Restored Indices Are Hidden (Step 7)

**Problem:** Restored indices inherited `index.hidden: true` from the `fm-clone-` source index. This caused them to be invisible to standard wildcard searches (`restored-logs-mytest-*/_search` returned 0 hits).

**Fix:** Added explicit un-hide step after restore, before querying:

```json
PUT restored-logs-mytest-default-2026.05.27-000001/_settings
{ "index": { "hidden": false } }

PUT restored-logs-mytest-default-2026.05.27-000002/_settings
{ "index": { "hidden": false } }
```

## Timing Note (Doc 3 landed in 000002)

The scenario called for a ~15s wait between documents. In practice, rollover processing for 000002 had not fully completed when Doc 3 was ingested, so Doc 3 landed in 000002 (alongside Doc 2) rather than in a new 000003. This is a timing edge case, not an ILM defect. The full lifecycle and restore still worked correctly. To ensure strict one-doc-per-shard separation in future runs, increase the inter-ingestion wait to 30s or poll until ILM explain shows the write index has changed before ingesting the next doc.
