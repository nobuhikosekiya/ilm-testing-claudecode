# ILM Restore Test — v8.19 SLM Storage Cost

## Summary

**Test intention:** Quantify the extra repository storage cost of retaining SLM snapshots
alongside ILM frozen snapshots. Specifically: does taking an SLM snapshot *after* force merge
(when ILM will snapshot the same single-segment state) add negligible storage overhead compared
to taking it *before* force merge?

**Result: Hypothesis confirmed.** For 4 out of 5 measured backing indices, the ILM frozen
snapshot added **0 incremental bytes** on top of the SLM Post-merge snapshot — both referenced
the exact same Lucene segment files. This means keeping an SLM Post-merge snapshot costs the
same total repository space as keeping no SLM snapshot at all: you pay once for the merged
segment (SLM Post-N pays it; ILM reuses it for free), and in return you hold an independent
recovery copy.

By contrast, SLM snapshots taken *before* force merge added ~9–13 KB of genuinely new blobs
per backing index (the multi-segment state that no other snapshot shares).

**Secondary finding:** An Elastic Cloud auto-SLM policy (`cloud-snapshot-...`) fired mid-run and
silently absorbed one Pre-N measurement, and ILM exhibited a stats-lag / phantom-rollover bug
that caused backing-index numbering to skip non-sequentially. Neither affected the core result.

## Findings

### F-1: SLM Post-merge snapshots deduplicate perfectly against ILM frozen snapshots (4/5 indices)

ILM incremental bytes = **0 B** for 4 out of 5 backing indices (000001, 000002, 000003, 000008).
When an SLM Post-N snapshot is taken AFTER force merging to 1 segment, and ILM's
`force_merge_index: false` means it snapshots the same already-merged data — the result is that
both SLM Post-N and ILM reference the exact same Lucene segment files. The ILM snapshot adds
**zero extra repository storage** on top of SLM Post-N.

For 000005, ILM incremental was 373 B (minor metadata divergence; see F-4).

### F-2: Keeping SLM Post-N adds zero net storage overhead over ILM-only

If you take SLM Post-N BEFORE ILM snapshots the same data:
- SLM Post-N pays the cost of the merged segment (5,078–5,182 B for these tiny 3-doc indices)
- ILM then costs 0 B incremental (blobs already present)
- Total = same as if no SLM snapshot had been taken

**Conclusion: SLM Post-merge snapshots are essentially free from a repository storage perspective.**
They provide an additional recovery path at no additional storage cost.

### F-3: ILM phantom rollovers when Lucene stats lag behind search count

**Pattern observed:** When docs are indexed and the index stats (`_cat/indices docs.count`)
show fewer documents than `_count` API returns, ILM at `check-rollover-ready` does not detect
the rollover condition. A `_refresh` or `_flush` call fixes the stat discrepancy, but then ILM
rolls over the corrected index AND creates multiple empty phantom write indices (one per 10s
poll cycle that was "waiting") in rapid succession before settling on the final write index.

**Observed phantom jumps:**
- 000003 rolled over → 000005 (000004 phantom, 1 extra rollover)
- 000005 refreshed → 000008 (000006, 000007 phantom, 2 extra rollovers)
- 000008 flushed → 000012 (000009, 000010, 000011 phantom, 3 extra rollovers)

The phantom backing indices are empty (0 docs) and pass through the lifecycle without issue.
They do not affect SLM/ILM snapshot correctness but inflate the backing index numbering.

**Root cause hypothesis:** ILM's `check-rollover-ready` loop accumulates pending evaluations
when the stat count lags. When the stat finally updates, all accumulated evaluations fire in
quick succession, each creating an additional rollover.

**Implication for scenario:** The scenario assumed clean sequential numbering (000001–000006).
Actual run used 000001, 000002, 000003, 000005, and 000008 as the 5 measured indices.

### F-4: Force merge was a no-op for 000005; ILM added minor metadata overhead

For backing index 000005, the force merge to 1 segment appeared to be a no-op:
- SLM Post-4 incremental = 0 B (identical blobs to Pre-4, meaning only 1 segment existed before merge)
- ILM incremental = 373 B (small overhead, possibly Lucene metadata or translog signature)

For 000005, the data appears to have landed in a single Lucene segment despite the 2s pause
between doc ingests, possibly due to the manual `_refresh` call consolidating all pending docs
into one segment before any separate refresh cycles fired.

### F-5: Cloud auto-SLM snapshot absorbs SLM Pre-N incremental for 000003

Elastic Cloud fired `cloud-snapshot-2026.05.28-c3emrak3slmriskao1mz1g` between SLM Post-2 and
SLM Pre-3. This cloud snapshot captured 000003's multi-segment state first, making SLM Pre-3
incremental = 0 B (the blobs were already in the repository). This distorts the Pre-3 measurement:
the true cost of pre-merge SLM for 000003 was absorbed by the cloud snapshot (9,078 B).

When measuring SLM Pre-N costs on Elastic Cloud, scheduled cloud-managed SLM snapshots can
silently absorb what appear to be "first capture" costs. In production this is beneficial but
makes cost attribution less transparent.

### F-6: SLM Post-N serves as viable recovery source after ILM snapshot expiry (Step 7 confirmed)

After deleting the ILM frozen snapshot for 000001 (simulation of retention expiry):
- Restore from SLM Post-1 (`approach-a-storage-cost-2026.05.28-ph54hzszseokmw4ncswcia`) succeeded
- All 3 original documents retrieved via `restored-default-logs-mytest-default-default-2026.05.28-000001`
- Restore required `index.hidden: false` setting (v8.19 behavior: inherited `index.hidden: true`)

## Notes for the scenario file

1. **ILM stats lag (critical):** `_cat/indices docs.count` (used by ILM for rollover evaluation)
   lags behind the actual doc count (`_count` API). ILM sits at `check-rollover-ready` waiting
   for stats to show max_docs. A `_refresh` or `_flush` call unblocks it but triggers phantom
   rollovers. The scenario should note this behavior and suggest a wait of 60–90s (or a `_refresh`
   call with a note about the phantom rollover side effect) after ingesting the 3rd doc.

2. **Phantom rollover mitigation:** The scenario should either:
   - Accept the phantom numbering and note it, OR
   - Not call `_refresh`/`_flush` manually and instead wait longer (90s+) for ILM auto-detection

3. **Cloud SLM interference:** On Elastic Cloud, auto-managed SLM snapshots (`cloud-snapshot-...`)
   fire every ~30 minutes and can silently absorb the "first capture" cost of SLM Pre-N snapshots.
   The scenario should note that Pre-N incremental bytes may appear as 0 if the cloud snapshot
   fires between rollover and the manual Pre-N execution. Run during a quiet window if possible.

4. **Namespace collision warning is correct:** Using `logs-mytest-default-default` (double-default)
   with priority 300 template worked correctly. The warning in the scenario header is appropriate.

5. **`force_merge_index: false` confirmed:** ILM did NOT re-merge the already-merged data; the
   ILM frozen snapshot referenced the same single-segment files that SLM Post-N captured.
   The `force_merge_index: false` setting is essential for the deduplication hypothesis.
