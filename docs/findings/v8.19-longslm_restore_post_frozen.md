# ILM Restore Test — v8.19 slm_backup

## Summary

**Intention**: Verify that SLM snapshots can serve as a reliable sole backup for ILM-managed data streams, even after the frozen tier and ILM's own searchable snapshots have been deleted. The test used `delete_searchable_snapshot: true` in the ILM policy so that ILM intentionally keeps no long-term copy — SLM (with 3-day retention and manual execution before each rollover) is the only recovery path.

**Result**: ✅ **CONFIRMED.** After ILM deleted all 3 backing indices and all 3 ILM snapshots, both target indices were successfully restored from SLM snapshots and returned the expected documents. SLM is a reliable backup when two conditions are met: (1) the snapshot is taken while the index still exists on the cluster, and (2) retention is long enough to outlive the ILM delete phase.

**Storage trade-off**: Using SLM with long retention carries an additional storage cost. ILM force-merges a backing index before taking its own searchable snapshot, producing a compact, merged copy. SLM snapshots are taken on a fixed schedule — often before ILM has run the force merge — so they capture the index in its pre-merge state (more segments, larger on disk). If both the early SLM snapshot (pre-merge) and a later SLM snapshot (post-merge, or after ILM has moved the index to frozen as a `partial-` mount) are within the retention window, the repository holds both representations of the same data, roughly doubling the storage for those indices until the older snapshot expires. See `docs/findings/v8.19-slm-storage-cost.md` for a detailed storage analysis.

## Findings

### Finding 1 — SLM RESTORE SUCCEEDED ✅
Backing indices `000001` and `000002` were successfully restored from SLM snapshots after:
- ILM had deleted the frozen-tier mounts
- ILM had deleted its own searchable snapshots (`delete_searchable_snapshot: true`)

Query `GET restored-slm-logs-mytest-*/_search` returned both docs:
- `"Log event 1 in backing index 1"` from `restored-slm-…-000001`
- `"Log event 2 in backing index 2"` from `restored-slm-…-000002`

**Conclusion**: SLM works as a reliable sole backup when (a) snapshots are taken while the index is still on the cluster (hot or frozen) and (b) retention outlives the ILM delete phase.

### Finding 2 — SLM captures hot-tier indices as bare `.ds-` names
SLM snapshot 1 was taken immediately after doc 1 was ingested, before ILM rolled it over. At that point, `000001` was still in the hot tier and the snapshot stored it as `.ds-logs-mytest-default-2026.05.28-000001` (bare name, no `partial-` prefix). This matches v8.19 documented behaviour for ILM snapshots and applies equally to SLM.

By the time SLM snapshot 3 was taken (~60 seconds later), `000001` had already entered the frozen tier; snapshot 3 stored it as `partial-.ds-logs-mytest-default-2026.05.28-000001`. Both name forms are restorable, but the `partial-` form requires a different `rename_pattern` (`.*partial-\.ds-(.+)` instead of `\.ds-(.+)`). Snapshots 1 and 2 captured `000001` as the bare `.ds-` form and were used for the restore in this test.

### Finding 3 — `delete_searchable_snapshot: true` deletes ILM snapshots in staggered steps
After the backing indices were deleted, the 3 ILM snapshots were not removed immediately. ILM deleted them one at a time at ~10–20 second intervals:
- `000001` snapshot: deleted at 03:11:44 (roughly 2m after the index was deleted)
- `000002` snapshot: deleted at 03:12:05
- `000003` snapshot: deleted at 03:12:47

All 3 were deleted within ~2 minutes of the index deletions. The ILM snapshot count dropped from 3 → 2 → 1 → 0 in three distinct steps across ~60 seconds.

### Finding 4 — ILM rolled backing indices to 000005 before the monitoring loop started
By the time Step 5 monitoring began at 03:09:25, the active write index was already `000005` (0 docs). Only `000003` (1 doc) and `000005` were visible in the hot tier; `000001` and `000002` were already deleted. This suggests ILM processed rollovers and frozen transitions for `000001`–`000004` within approximately 24 seconds of the last ingest (03:09:01). The existence of `000004` and `000005` beyond the 3 explicitly ingested indices is consistent with ILM rolling over `000003` (max_docs: 1 hit) → creating `000004`, and a subsequent ILM evaluation creating `000005`. ILM snapshots for `000001`, `000002`, and `000003` were all created, but no ILM snapshot was observed for `000004`, suggesting `000004` was deleted before ILM could reach the frozen phase for it (or it was an empty write index that was rolled over immediately).

### Finding 5 — Restored indices are hidden; `expand_wildcards=all` required to see them
After the SLM restore was accepted (`"accepted": true`), `GET _cat/indices/restored-slm-*?v` returned empty results. The indices were present but hidden (`index.hidden: true` inherited from the original backing index). `GET _cat/indices/restored-slm-*?v&expand_wildcards=all` revealed them with 1 doc each. A subsequent `PUT <index>/_settings { "index": { "hidden": false } }` was required before a wildcard `GET restored-slm-*/_search` would work. This is the same v8.19 behaviour noted in `host_frozen_delete_keepsnapshot.txt` and `normal_slm_usage_with_frozen.txt`.

## Notes for the scenario file

1. **Monitoring loop termination**: The scenario's Step 5 poll condition checks for `000001` returning 404. In this run, `000001` was already deleted before the monitoring loop started (i.e. within ~24 seconds of the last ingest). The poll loop exited on its first iteration. This is expected given how fast ILM processed everything at poll_interval: 10s; the condition still works correctly.

2. **Investigate 000004/000005 numbering**: The scenario expects 3 backing indices (000001–000003) plus a current write index (000004). In this run, the write index was 000005 by monitoring time. The cause is unclear — possible ILM transition artefact or a rapid rollover of an empty write index. Worth a follow-up investigation. Does not affect test outcome.

3. **SLM snapshot 3 stores 000001 as `partial-`**: The scenario comments say "Because we ran _execute before each rollover (Step 4), the hot-tier name (.ds-...) is expected here." This is true for snapshots 1 and 2 but NOT snapshot 3. By the time of snapshot 3, `000001` had already entered the frozen tier (~60 seconds after ingest). The scenario should clarify that only snapshot 1 is guaranteed to capture `000001` as a hot-tier index; later snapshots may capture it as `partial-`. The restore used snapshots 1 and 2 (both hot-tier), so no issue in practice, but the comment should be updated.

4. **No `index.hidden` note in Step 6**: The scenario's restore section does not proactively warn that `GET _cat/indices/restored-slm-*` will return empty results due to hidden indices. Consider adding: "Note: the restored indices inherit `index.hidden: true`. Use `?expand_wildcards=all` to verify they exist before proceeding to Step 7."
