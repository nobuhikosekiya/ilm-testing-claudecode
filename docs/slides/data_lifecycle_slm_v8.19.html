Frozen-Tier Backing Index Recovery — What Survives & What Doesn't

Hot Tier → Frozen Tier → Deleted from cluster → recover via ILM Snapshot ✓ or SLM Snapshot ?
Key question: Once ILM deletes a Frozen-tier backing index and SLM retention rotates past the snapshots that captured it — is the data gone?
→ This scenario proves SLM is NOT a reliable recovery path; ILM-managed snapshots are.

ILM Policy — approach_a_ds_policy

Governs how each backing index of the data stream moves through tiers and is ultimately deleted.

① Hot
rollover: max_docs: 1
As soon as one document is ingested, ILM rolls over — creating the next backing index. The current one moves toward Frozen.
→
② Frozen
min_age: 0m
searchable_snapshot:
repo: found-snapshots
ILM immediately takes a searchable snapshot of the backing index, mounts it as partial-… on the frozen tier, and removes the original shards.
→
③ Delete
min_age: 3m
delete_searchable_snapshot: false ⚠️
After 3 minutes the frozen mount is removed from the cluster — but the underlying ILM snapshot is kept in found-snapshots.
This is what enables restore later.
Critical setting: "delete_searchable_snapshot": false
This tells ILM: "unmount the frozen index, but do not delete the snapshot from the repository."
Without this flag set to false, the snapshot would be deleted along with the index — and recovery would be impossible.

What ILM snapshots look like (v8.19)

Named: <YYYY.MM.DD>-<index-name>-<policy>-<uuid>

In v8.x, ILM force-merges the original backing index in place — so the snapshot contains the raw .ds-… name (no fm-clone- prefix).

Contrast with v9.2+: ILM clones to fm-clone-<uuid>-, snapshots the clone — different restore rename_pattern needed.

SLM Policy — approach_a_slm_policy

A cluster-wide snapshot policy, manually executed at precise moments to demonstrate its limitations.

Configuration

name: <approach-a-slm-{now/d}> schedule: "0 0 0 1 1 ? 2099" ← never fires automatically repository: found-snapshots config: indices: "*" ← captures everything partial: true ← survives rollover transitions retention: expire_after: 30d min_count: 1 max_count: 3 ← ⚠️ THIS IS THE DANGER
⚠️ The max_count: 3 Trap

SLM will only ever keep the 3 newest snapshots.
Once you run a 4th execution, the oldest is automatically deleted by retention.
If that oldest snapshot was the only one that captured a now-deleted backing index — that data is gone.

Why partial: true matters

During a rollover, an index may be momentarily in transition. partial: true lets the snapshot succeed even if some shards are temporarily unavailable.

The core risk: SLM snapshots are cluster-wide point-in-time captures. They don't know about — or care about — the ILM lifecycle of individual backing indices. Once ILM deletes a backing index and SLM retention rotates past the snapshots that captured it, the data is unrecoverable via SLM.

Restore command pattern (v8.19)

POST _snapshot/found-snapshots/<ilm_snapshot_name>/_restore { "indices": ".ds-logs-mytest-default-2026.05.27-000001", "rename_pattern": "\\.ds-(.+)", ← strips ".ds-" prefix "rename_replacement": "restored-$1", ← renames to "restored-…" "index_settings": { "index.lifecycle.name": null, ← detach from ILM "index.routing.allocation.include._tier_preference": "data_hot" } }
⚠️ Remember: Restored indices inherit index.hidden = true from the backing index. Run PUT restored-…/_settings {"index": {"hidden": false}} before wildcard searches will find them.

ILM vs SLM — Reliability Comparison for Frozen-Tier Recovery

Property ILM Snapshots
delete_searchable_snapshot: false		 SLM Snapshots
max_count: 3 (or any limit)
Snapshot scope One snapshot per backing index, tied to its lifecycle Cluster-wide, captures all indices at one moment
Retention model Kept until explicitly deleted (e.g., cleanup script) Count/time bounded — oldest auto-deleted
Capture timing Taken by ILM at the exact moment of the Frozen-phase transition Taken at arbitrary scheduled/manual times — may miss windows
Recovery window Unlimited — survives ILM delete phase Narrow — closes once retention rotates past the capturing snapshot
v8.19 index name in snapshot .ds-logs-mytest-…-000001 (raw name, no fm-clone) .ds-logs-mytest-…-000001 (same, while index exists)
Restore rename_pattern "\\.ds-(.+)" → "restored-$1" "\\.ds-(.+)" → "restored-slm-$1"
Reliability verdict ✓ Reliable — purpose-built for this use case ✗ Unreliable — will eventually lose the data
v9.2+ note: In v9.2+, ILM creates a fm-clone-<uuid>-.ds-… copy before snapshotting. The ILM snapshot contains the clone name — not the raw .ds-… name. Use rename_pattern: ".*\\.ds-(.+)" to strip the clone prefix during restore. SLM behaviour and its retention problem are identical across versions.

Key Takeaways

✅ ILM-managed snapshots are the correct recovery path for Frozen-tier data
  • Set "delete_searchable_snapshot": false in the Delete phase
  • ILM snapshot is taken at Frozen-phase entry — before deletion
  • Survives indefinitely until you explicitly clean up
  • Use restore_index_v8.19.sh to recover a specific backing index by name
❌ SLM snapshots are NOT reliable for Frozen-tier backing index recovery
  • Retention (max_count / expire_after) rotates out snapshots unpredictably
  • New snapshots taken after ILM deletion don't contain the deleted index
  • The recovery window shrinks with every SLM execution
  • With any realistic SLM schedule, the window closes before it's needed

Architecture Recommendation

ILM Delete phase: "delete": { "min_age": "...", "actions": { "delete": { "delete_searchable_snapshot": false ← MUST be false } } } Cleanup: Run cleanup_old_snapshots_v8.19.sh (e.g., delete ILM snaps older than 2 years) — on your own schedule, not SLM's SLM: Use for cluster-wide disaster recovery (full cluster restore, node loss) — NOT for individual index recovery after Frozen-tier deletion
Remember: After restoring, set index.hidden = false on the restored index, or wildcard searches won't find it.
